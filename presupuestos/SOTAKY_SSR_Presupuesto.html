1. Contexto del Problema

⚠️ Situación Actual

Actualmente el admin.page (Angular/Ionic) envía todo el HTML/JS al cliente, incluyendo secciones de Equipo, Alumnos, Profesores, etc. Un usuario con conocimientos técnicos podría inspeccionar el código fuente y ver la estructura completa del panel, incluso si no tiene permisos.

El backend (NestJS) actualmente no tiene guards ni middleware de autenticación/autorización. Esto implica que el trabajo no es solo "renderizar HTML desde el server", sino construir toda la capa de seguridad desde cero.

2. Alcance del Trabajo — 3 Fases

Fase 1 — Backend: Sistema de autenticación y autorización

1. JWT Auth Guard Crear JWT Auth Guard en NestJS (validar tokens en cada request)
2. Roles Guard + Decorador @Roles() Proteger endpoints por rol de usuario
3. Protección de Endpoints Proteger todos los endpoints del admin (cursos, alumnos, profesores, equipo, videos, materiales) con guards de rol
4. Endpoint GET /admin/config Devuelve las secciones permitidas según el rol del usuario autenticado

Fase 2 — Backend: Endpoint de configuración dinámica

5. Estructura de navegación dinámica Endpoint que retorne secciones permitidas por rol (admin, moderator, editor, viewer)
6. Migración de localStorage al backend Conectar la gestión de equipo a una tabla de permisos real
7. Arquitectura de permisos granular Diseñar estructura flexible para soportar futuras extensiones

Fase 3 — Frontend: Renderizado condicional seguro

8. Admin Guard en Angular Validar rol antes de cargar la ruta
9. Consulta dinámica al backend Modificar admin.page.ts para consultar qué secciones puede ver el usuario
10. Lazy loading por sección Cargar dinámicamente solo las secciones autorizadas
11. Eliminación de lógica localStorage Migrar toda la lógica de permisos al backend
12. Pruebas de seguridad Verificar que un usuario sin permisos no pueda acceder a endpoints ni ver secciones
13. Hardening Ajustes, correcciones y hardening de la arquitectura base
